# 多 Agent 调度学习项目计划

## 目标

通过一个循序渐进的实践项目，理解多 Agent 系统中的角色拆分、任务调度、上下文传递、结果汇总、冲突处理与质量评估。最终完成一个“小型多 Agent 任务协作平台”：用户输入一个复杂任务，系统自动拆解任务，分配给不同 Agent，并汇总输出最终结果。

## 推荐技术栈

- 语言：Python 或 TypeScript，优先选择你更熟悉的语言
- Agent 框架：先手写最小调度器，再尝试 LangGraph、AutoGen、CrewAI 等框架
- 模型调用：OpenAI API 或兼容 OpenAI SDK 的本地/云端模型
- 数据存储：从 JSON 文件开始，后续可升级为 SQLite
- 可视化：先用命令行日志，后续可增加简单 Web UI

## 项目最终形态

系统包含以下核心模块：

- `User Interface`：接收用户任务，展示执行过程和最终结果
- `Planner Agent`：分析任务并拆分子任务
- `Dispatcher`：根据任务类型、优先级和依赖关系分配 Agent
- `Worker Agents`：执行具体子任务，例如检索、编码、写作、评审
- `Reviewer Agent`：检查结果质量，指出问题或要求返工
- `Memory`：保存任务、上下文、执行记录和中间结果
- `Aggregator`：合并多个 Agent 的输出，生成最终答案

## 阶段 1：单 Agent 基础任务执行

### 学习目标

- 理解 Agent 的基本循环：接收任务、推理、执行、返回结果
- 学会记录输入、输出、错误和执行耗时
- 为后续多 Agent 调度建立最小可运行基础

### 实践任务

1. 创建一个 `BasicAgent`，支持接收文本任务并返回结果。
2. 给 Agent 增加统一的 `run(task)` 方法。
3. 为每次执行生成 `task_id`、开始时间、结束时间和状态。
4. 将执行记录保存到本地 JSON 文件。

### 验收标准

- 能从命令行输入一个任务并得到 Agent 输出
- 每次执行都有可追踪的日志
- Agent 失败时能返回明确错误信息，而不是直接崩溃

## 阶段 2：任务拆解与计划生成

### 学习目标

- 理解复杂任务为什么需要拆分
- 学会把自然语言任务转换为结构化子任务
- 为 Dispatcher 提供可执行的任务队列

### 实践任务

1. 创建 `PlannerAgent`，负责把用户任务拆成多个子任务。
2. 设计子任务数据结构，例如：

```json
{
  "id": "task-001",
  "title": "收集资料",
  "description": "查找多 Agent 调度相关概念",
  "type": "research",
  "depends_on": [],
  "priority": 1,
  "status": "pending"
}
```

3. 支持任务依赖关系，例如“先检索资料，再写总结，再评审”。
4. 将计划保存为 `plan.json`。

### 验收标准

- 输入一个复杂任务后，能生成 3 到 6 个结构化子任务
- 子任务包含类型、依赖、优先级和状态
- 子任务之间的执行顺序清晰可解释

## 阶段 3：多 Worker Agent 分工

### 学习目标

- 理解 Agent 角色划分
- 学会根据任务类型分配不同 Agent
- 初步实现一对多的调度模式

### 实践任务

1. 创建多个 Worker Agent：
   - `ResearchAgent`：负责资料整理
   - `WriterAgent`：负责内容生成
   - `CodeAgent`：负责代码或伪代码实现
   - `ReviewAgent`：负责质量检查
2. 创建 `Dispatcher`，根据 `task.type` 选择合适的 Agent。
3. 将每个子任务的输出保存到 `outputs/{task_id}.md`。
4. 在日志中记录“任务由哪个 Agent 执行”。

### 验收标准

- 不同类型任务能分配给不同 Agent
- 每个 Agent 有清晰职责边界
- 输出结果可以追踪到具体 Agent 和子任务

## 阶段 4：依赖调度与执行顺序控制

### 学习目标

- 理解任务依赖图
- 学会处理不能立即执行的任务
- 实现简单的 DAG 调度逻辑

### 实践任务

1. 实现任务状态流转：
   - `pending`
   - `ready`
   - `running`
   - `completed`
   - `failed`
2. 只有依赖任务全部完成后，子任务才能进入 `ready`。
3. 添加一个调度循环：
   - 找到所有 `ready` 任务
   - 分配给对应 Agent
   - 写入结果
   - 更新状态
4. 支持失败任务阻塞依赖它的后续任务。

### 验收标准

- 任务不会在依赖未完成时提前执行
- 调度日志能展示完整执行顺序
- 任意任务失败时，后续依赖任务不会错误执行

## 阶段 5：上下文传递与结果汇总

### 学习目标

- 理解多 Agent 协作中的上下文传递
- 学会控制上下文大小，避免无关信息污染结果
- 实现多个子任务结果的统一汇总

### 实践任务

1. 为每个子任务构建上下文：
   - 用户原始任务
   - 当前子任务描述
   - 依赖任务输出摘要
2. 创建 `AggregatorAgent`，负责合并所有已完成子任务结果。
3. 为每个输出生成简短摘要，减少传递给后续 Agent 的上下文长度。
4. 最终生成 `final_report.md`。

### 验收标准

- 后续任务能使用前置任务结果
- 最终报告结构清晰，不只是简单拼接
- 上下文中不包含大量无关历史记录

## 阶段 6：评审、返工与质量控制

### 学习目标

- 理解多 Agent 中的反馈闭环
- 学会用 Reviewer Agent 检查 Worker 输出
- 初步实现自动返工机制

### 实践任务

1. 为每个 Worker 输出添加评审步骤。
2. `ReviewAgent` 输出结构化评审结果：

```json
{
  "approved": false,
  "issues": ["缺少示例", "结论不够明确"],
  "suggestions": ["增加一个最小可运行例子", "补充优缺点对比"]
}
```

3. 如果 `approved=false`，将任务退回给原 Agent 修改。
4. 设置最大返工次数，例如最多 2 次，避免无限循环。

### 验收标准

- 系统能识别低质量输出
- Worker Agent 能基于评审意见改进结果
- 返工次数受控，不会无限运行

## 阶段 7：并行调度与资源限制

### 学习目标

- 理解哪些任务可以并行，哪些任务必须串行
- 学会控制并发数和执行队列
- 初步理解多 Agent 系统中的资源调度

### 实践任务

1. 找出所有没有相互依赖的 `ready` 任务。
2. 使用线程、异步任务或任务队列并行执行。
3. 增加并发限制，例如同一时间最多运行 3 个 Agent。
4. 增加超时机制，超过指定时间则标记为 `failed`。

### 验收标准

- 无依赖任务可以并行执行
- 并发数不会超过配置上限
- 超时任务会被明确记录

## 阶段 8：记忆系统与经验复用

### 学习目标

- 理解短期记忆和长期记忆的区别
- 学会保存可复用的执行经验
- 为 Agent 提供历史上下文检索能力

### 实践任务

1. 创建 `MemoryStore`，记录：
   - 原始任务
   - 拆解计划
   - Agent 输出
   - 评审结果
   - 最终报告
2. 支持根据关键词检索历史任务。
3. 在 Planner 阶段参考相似历史任务。
4. 可选：使用向量数据库做语义检索。

### 验收标准

- 能查看历史任务执行记录
- 新任务能复用相似任务的拆解经验
- 记忆不会无筛选地塞进所有上下文

## 阶段 9：可观测性与调试面板

### 学习目标

- 理解多 Agent 系统为什么需要可观测性
- 学会定位调度错误、上下文错误和输出质量问题
- 提升系统可调试性

### 实践任务

1. 为每次任务执行生成 trace。
2. 记录每个 Agent 的：
   - 输入
   - 输出
   - 执行时间
   - token 消耗或估算成本
   - 状态变化
3. 创建一个简单的 Markdown 或 HTML 报告展示执行过程。
4. 可选：实现 Web UI 展示任务 DAG 和执行状态。

### 验收标准

- 能回答“为什么这个任务由这个 Agent 执行”
- 能定位失败发生在哪个子任务
- 能看到完整的执行链路

### 任务示例

> 请调研 LangGraph、AutoGen 和 CrewAI 的多 Agent 调度机制，比较它们的优缺点，并给出一个适合初学者的学习路线。

### 期望执行流程

1. `PlannerAgent` 拆分任务。
2. `ResearchAgent` 分别整理三个框架的信息。
3. `WriterAgent` 生成对比表和学习路线。
4. `ReviewAgent` 检查是否覆盖关键维度。
5. 如果质量不足，触发返工。
6. `AggregatorAgent` 生成最终报告。

- 最终报告包含背景、对比表、优缺点和学习建议
- 每个结论可以追踪到对应子任务输出
- 调度日志完整展示 Agent 协作过程

## 建议学习节奏

| 周期 | 学习内容 | 交付物 |
| --- | --- | --- |
| 第 1 周 | 单 Agent、日志、任务结构 | `BasicAgent` 和执行日志 |
| 第 2 周 | Planner、任务拆解、Dispatcher | `plan.json` 和 Agent 分发 |
| 第 3 周 | DAG 调度、上下文传递、汇总 | `final_report.md` |
| 第 4 周 | Review、返工、并行调度 | 自动评审与并发执行 |
| 第 5 周 | Memory、trace、调试面板 | 可观测执行报告 |
| 第 6 周 | 综合挑战与框架对比 | 完整多 Agent 项目 |

## 每阶段复盘问题

- 当前阶段解决了多 Agent 调度中的哪个问题？
- Agent 的职责是否足够清晰？
- 调度器是否掌握了任务状态和依赖关系？
- 上下文是否足够但不过量？
- 失败、超时和低质量输出是否能被处理？
- 结果是否能追踪到具体 Agent 和子任务？

## 后续扩展方向

- 增加人类确认节点，例如重要步骤需要人工审批
- 支持动态创建 Agent，而不是固定角色列表
- 增加成本控制策略，例如限制 token、限制重试次数
- 增加工具调用能力，例如搜索、文件读写、代码执行
- 将调度器升级为图结构框架，例如 LangGraph
- 对比手写调度器与现成框架的复杂度和可维护性
